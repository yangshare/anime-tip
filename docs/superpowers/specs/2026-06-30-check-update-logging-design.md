# 检查更新日志增强 — 设计规格

- **日期**：2026-06-30
- **范围**：`internal/scheduler/scheduler.go` 的 `CheckUpdates()` 方法
- **状态**：已批准，待编写实现计划

## 背景与问题

当前 `CheckUpdates()` 只有零星几条 `log.Println/Printf`，输出到 stdout。存在两个观测盲区：

1. **不知道何时做过检查**——开始日志没有关注列表规模，结束时无任何汇总，无法从日志还原一次完整检查。
2. **不知道哪个环节失败**——数据源接口不通（`GetAnimeDetail` 失败）、通知推送不通（ServerChan 失败）时，错误信息要么被 `continue` 吞掉只留一行、要么直接 `return err` 上抛而本地无痕迹，事后排查困难。

## 目标

每次检查能从日志里完整还原：什么时候查了、查了多少部、哪些环节出错、总耗时多少。

## 非目标（YAGNI）

- 不引入结构化日志库（zap/logrus 等），保持全程标准 `log` 包风格一致。
- 不写文件轮转，依赖外部收集（docker logs / journald）。
- 不持久化到数据库、不加前端历史页面（已评估并否决）。

## 设计

仅修改 `internal/scheduler/scheduler.go`，保持现有 `[scheduler]` 前缀和 `log` 包风格。在以下节点补充/修改日志：

| 节点 | 现状 | 改进 |
|------|------|------|
| 开始检查 | 仅一句话 | 补关注列表数量：`开始检查动漫更新，共 N 部` |
| 单部抓取失败 | 已有 log | 保持，已含动漫名/vod_id/错误 |
| 单部抓取成功但无变化 | 无 | **不记**（避免噪音） |
| 发现更新汇总 | 无 | `发现 X 部有更新：《..》《..》` |
| 推送 ServerChan | 仅成功后一行 | 推送失败时在 `return err` 前补一条明确失败 log |
| 检查结束 | 无 | 补结束日志 + 总耗时：`本次检查完成，耗时 X` |

### 取舍

1. **不记录"无变化"的成功项**：20 部动漫会刷屏，违背"想知道问题"的初衷。只在失败和有更新时打点。
2. **接口/推送失败的可见性**：`GetAnimeDetail` 失败已 `continue` + log，保留；ServerChan 推送失败在 `return err` 前补 log，确保本地留痕。
3. **耗时**用 `time.Since()` 计算，便于判断某次检查是否因数据源变慢而拖长。
